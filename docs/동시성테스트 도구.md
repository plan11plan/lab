# 동시성 테스트로 레이스 컨디션을 재현하는 방법

## 서론

### 동시성 충돌

컴퓨터 세계에 진짜 "동시(simultaneous)"라는 건 없다. CPU 코어가 아무리 많아도 OS 스케줄러 관점에서는 여러 스레드의 명령어가 시간축 위에서 잘게 쪼개져 끼워 넣어지는 인터리빙(interleaving)으로 처리된다. "동시 요청"이라고 부르는 것도 실제로는 "거의 같은 순간에 인터리빙되는 요청들"에 가깝다.

그렇다면 충돌은 무엇인가. 인터리빙 자체는 문제를 일으키지 않고, **트랜잭션 임계영역(critical section)이 겹칠 때**에만 문제가 된다. 임계영역이란 공유 자원을 읽고 판단하고 쓰는 구간을 가리킨다. 재고 차감을 예로 들면 재고 조회 → 0보다 큰지 확인 → 1 차감 세 단계 전체가 하나의 임계영역에 해당한다.

```
T1: |--조회--확인--차감--|
T2:        |--조회--확인--차감--|
                ↑
       T1이 차감하기 전에 T2가 조회 → 둘 다 같은 값 보고 1씩 차감
```

임계영역에 머무는 시간이 곧 다른 스레드의 명령어가 끼어들 수 있는 시간이므로, **임계영역이 길수록 충돌 확률이 올라간다.** 락을 보유하는 시간이 길수록 다른 트랜잭션이 같은 자원을 건드릴 가능성이 커지고, **락 점유 시간 = 충돌 윈도우**라고도 볼 수 있다.

### 동시성 테스트가 하는 일

그렇다면 동시성 테스트가 해야 할 일도 정해진다. **임계영역이 끝나기 전에 여러 요청을 "대기 상태"로 정렬시키는 것**이다. 자연스럽게 도착하는 요청에 기대지 않고, 모든 스레드를 출발선에 세워두고 동시에 출발 신호를 쏘는 것에 가깝다. 한 스레드가 임계영역을 통과하는 동안 다른 스레드들도 같은 임계영역에 도달해 있어야, 비로소 race가 재현된다.

그러므로 동시성 테스트의 핵심 질문은 결국 **"동시"를 얼마나 정밀하게 보장하는가**로 모인다. 출발선 정렬이 어긋나면 일부 스레드는 이미 임계영역을 빠져나간 뒤에 다른 스레드가 도착하므로, 같은 코드를 같은 횟수로 돌려도 race가 잡히지 않는다. 동시성 테스트 도구의 차이는 결국 이 정렬 정밀도와 코드 복잡도 사이의 균형으로 모인다.

### 자바에서 쓰는 세 가지 방법

자바에서 동시성 테스트를 쓸 때 흔히 거론되는 방법은 세 가지다.

1. `ExecutorService` + 단일 `CountDownLatch`
2. 3-Latch 패턴 (`ready` / `start` / `done`)
3. `CompletableFuture`

이 글은 세 방법을 **사용법 → 작동 방식 → 특징** 순서로 정리한다.

### 미리 살펴볼 도구 — Counter / SafeCounter

본문에서 race가 발생할 공유 자원으로 사용할 단순한 카운터를 미리 정의한다.

```java
public class Counter {
    private int count = 0;
    public void increment() {
        int snapshot = count;   // R
        Thread.yield();         // R과 W 사이에 다른 스레드가 끼어들 여지를 둔다
        count = snapshot + 1;   // W
    }
    public int get() { return count; }
}

public class SafeCounter {
    private int count = 0;
    public synchronized void increment() {
        int snapshot = count;
        Thread.yield();
        count = snapshot + 1;
    }
    public synchronized int get() { return count; }
}
```

`Counter`는 increment를 일부러 R → yield → W로 분해해 임계영역을 살짝 넓혀 둔다. 실제 코드의 임계영역에는 비즈니스 로직·DB 호출·로깅 등이 끼어 자연히 폭이 생기므로, `count++` 한 줄짜리 코드보다 이 형태가 현실 시나리오에 더 가깝다. `SafeCounter`는 같은 로직을 `synchronized`로 감싸 race를 차단한다.

세 패턴 모두 `Counter`를 30번 증가시키는 동일한 시나리오에서 비교한다.

## 안티패턴 먼저: 단순 for문이 동시성 테스트가 아닌 이유

> ⚠️ **단순 for문은 안티패턴 시연이다.** 동시성 테스트 자리에 흔히 등장하지만 실제로는 race를 재현하지 못한다. 본문 세 패턴으로 넘어가기 전에 무엇이 빠졌는지 짚어둔다.

```java
@Test
void unsafe_counter_passes_with_for_loop() {
    Counter counter = new Counter();
    int callCount = 30;

    for (int i = 0; i < callCount; i++) {
        counter.increment();
    }

    assertThat(counter.get()).isEqualTo(callCount);
}
```

**실측 결과 (5회)** — 5회 모두 `actual=30`으로 통과한다.

문제는 통과한다는 점에 있다. for문은 한 increment가 끝난 뒤에야 다음 increment가 시작되므로 임계영역이 시간축에서 겹치지 않는다. 따라서 이 테스트는 "직렬에서 30번 증가하면 30이 된다"는 사실만 증명할 뿐, 동시 호출 상황인지 알기 어렵다.

동시성 테스트를 제대로 짜려면 다음 세 가지를 모두 만족해야 한다.

- **시작 정렬** — 여러 스레드를 같은 시점에 임계영역에 진입시킨다.
- **완료 동기화** — 모든 스레드가 끝난 뒤에 검증한다.
- **결과 검증** — 최종 상태가 기대값과 일치하는지 확인한다.

세 가지 방법은 결국 이 세 조건을 코드로 구현하는 서로 다른 방식이다.

## 방법 1. ExecutorService + 단일 CountDownLatch

가장 흔히 보는 구성으로, 스레드 풀을 만들고 latch 하나로 일제 출발을 신호한 뒤 다른 latch 하나로 완료를 기다린다.

### 코드

```java
@Test
void single_latch_test() throws InterruptedException {
    Counter counter = new Counter();
    int threadCount = 30;
    ExecutorService executor = Executors.newFixedThreadPool(32);

    CountDownLatch start = new CountDownLatch(1);          // 출발 신호용
    CountDownLatch done = new CountDownLatch(threadCount); // 완료 동기화용

    for (int i = 0; i < threadCount; i++) {
        executor.submit(() -> {
            try {
                start.await();          // 출발 신호 기다림
                counter.increment();    // 임계영역
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            } finally {
                done.countDown();       // "나 끝났다"
            }
        });
    }

    start.countDown();                  // 출발!
    done.await(10, TimeUnit.SECONDS);   // 전원 완료까지 대기
}
```

### 작동 방식

1. `submit`으로 던져진 태스크들은 `start.await()`에서 대기 상태로 멈춘다.
2. 메인 스레드가 `start.countDown()`을 호출하면 대기 중이던 태스크들이 풀려난다.
3. 각 태스크는 increment를 끝낸 뒤 `done.countDown()`으로 종료를 알린다.
4. 메인 스레드는 `done.await()`로 모든 태스크의 완료를 기다린 뒤 검증한다.

핵심은 출발 신호용 latch 하나로 모든 워커를 같은 시점에 출발시킨다는 점이다. 비록 워커가 풀에서 잡히는 시점이 살짝 차이 날 수 있지만, `start.await()`에 이미 도달한 워커들은 일제히 풀려난다.

### 특징

| 항목 | 내용 |
|---|---|
| 코드 복잡도 | 낮음 |
| 시작 정렬 정밀도 | 보통 (큐잉으로 일부 스레드가 늦게 합류 가능) |
| 적합한 상황 | 빠르게 race 가능성을 확인, 일반적인 lost update 재현 |
| 주의할 점 | 풀 크기를 너무 작게 잡으면 큐잉이 강해져 직렬에 가까워질 수 있고, 너무 크게 잡으면 그 자체로는 큰 영향이 없다 |

**실측 (Counter 30번, 풀=32, 5회)**

| Run | actual | lost |
|---|---|---|
| 1 | 23 | 7 |
| 2 | 23 | 7 |
| 3 | 16 | 14 |
| 4 | 17 | 13 |
| 5 | 8 | 22 |

5회 모두 race가 잡혔고, 평균 약 12.6건의 lost update가 발생한다.

## 방법 2. 3-Latch 패턴 (ready / start / done)

단일 Latch 방식의 빈틈인 "워커가 진짜 출발선에 도착했는지 모른다"는 점을 보완한다. latch 하나를 더 추가해 도착 확인 단계를 명시한다.

### 코드

```java
@Test
void three_latch_test() throws InterruptedException {
    Counter counter = new Counter();
    int threadCount = 30;
    ExecutorService executor = Executors.newFixedThreadPool(threadCount);

    CountDownLatch ready = new CountDownLatch(threadCount); // 도착 확인용
    CountDownLatch start = new CountDownLatch(1);            // 출발 신호용
    CountDownLatch done  = new CountDownLatch(threadCount);  // 완료 동기화용

    for (int i = 0; i < threadCount; i++) {
        executor.submit(() -> {
            ready.countDown();          // "나 출발선 도착"
            try {
                start.await();          // 출발 신호 기다림
                counter.increment();    // 임계영역
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            } finally {
                done.countDown();
            }
        });
    }

    ready.await();                      // 전원 도착할 때까지 대기
    start.countDown();                  // 진짜 일제 출발
    done.await(10, TimeUnit.SECONDS);
}
```

### 작동 방식

각 워커가 실행되자마자 `ready.countDown()`으로 "나 도착했다"를 신고하고 `start.await()`에서 멈춘다. 메인 스레드는 `ready.await()`로 모든 워커의 도착을 확인한 뒤에야 `start.countDown()`을 호출하므로, 출발 신호가 떨어지는 시점에는 모든 워커가 확실히 `start.await()` 직전에 정렬되어 있다.

세 latch가 좋은 동시성 테스트의 세 조건에 각각 매핑된다.

| latch | 매핑되는 조건 |
|---|---|
| `ready` | 시작 정렬 (출발선 도착 확인) |
| `start` | 시작 정렬의 완성 (일제 출발 신호) |
| `done` | 완료 동기화 |

여기서는 풀 크기를 `threadCount`와 같게 잡는다. 큐잉 없이 모든 워커가 동시에 활성화돼야 `ready.await()`이 의미를 갖기 때문이다.

### 특징

| 항목 | 내용 |
|---|---|
| 코드 복잡도 | 높음 (latch 3개 관리) |
| 시작 정렬 정밀도 | 가장 높음 |
| 적합한 상황 | 임계영역이 좁고 정밀한 정렬이 필요한 회귀 테스트, DB 트랜잭션처럼 ms 단위 임계영역의 검증 |
| 주의할 점 | 풀 크기를 `threadCount`보다 작게 잡으면 일부 워커가 큐에서 대기하게 되어 `ready` latch의 의미가 약해진다 |

**실측 (Counter 30번, 풀=30, 5회)**

| Run | actual | lost |
|---|---|---|
| 1 | 13 | 17 |
| 2 | 24 | 6 |
| 3 | 16 | 14 |
| 4 | 25 | 5 |
| 5 | 18 | 12 |

5회 모두 race가 잡혔고, 평균 약 10.8건의 lost update가 발생한다. 변동 폭은 단일 Latch보다 약간 좁다.

## 방법 3. CompletableFuture

가장 간결한 코드. 출발 신호용 latch 없이 `runAsync`로 던지자마자 실행하고, `allOf().join()`이 완료 동기화만 담당한다.

### 코드

```java
@Test
void completable_future_test() {
    Counter counter = new Counter();
    int threadCount = 30;
    ExecutorService executor = Executors.newFixedThreadPool(32);

    List<CompletableFuture<Void>> futures = new ArrayList<>();
    for (int i = 0; i < threadCount; i++) {
        futures.add(CompletableFuture.runAsync(counter::increment, executor));
    }
    CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).join();
}
```

### 작동 방식

`runAsync`는 태스크를 풀에 던지자마자 워커가 잡는 즉시 실행을 시작한다. 별도의 출발 신호가 없으므로 먼저 큐잉된 태스크가 먼저 출발하고, `allOf(...).join()`이 모든 태스크의 완료를 기다린다.

코드는 가장 간결하지만, 워커가 태스크를 잡는 순서대로 작업이 시작되므로 시작 정렬이 보장되지 않는다. 태스크가 풀에서 순차적으로 빠지는 동안 앞서 시작된 태스크가 이미 increment를 마쳐버리면, 임계영역이 겹칠 시간축이 좁아진다.

### 특징

| 항목 | 내용 |
|---|---|
| 코드 복잡도 | 가장 낮음 |
| 시작 정렬 정밀도 | 없음 |
| 적합한 상황 | 멱등 연산의 병렬 실행, 읽기 위주 시나리오, 가벼운 부하 테스트, 비동기 파이프라인 검증 |
| 주의할 점 | 시작 정렬이 없어 짧은 임계영역에서는 race가 잘 재현되지 않을 수 있다 — 신뢰성 있는 race 검출에는 latch 기반 패턴이 더 적합하다 |

**실측 (Counter 30번, 풀=32, 5회)**

| Run | actual | lost |
|---|---|---|
| 1~5 | 30 | 0 |

5회 모두 race가 잡히지 않았다. 시작 정렬이 없는 데다 풀 크기가 `threadCount`보다 크기 때문에, 태스크가 즉시 워커에 분배되어 일부는 다른 태스크가 시작되기 전에 increment를 마친다. 결국 임계영역끼리 시간축에서 겹칠 기회가 줄어 race가 표면으로 드러나지 않는다.

> 💡 CompletableFuture로 race 검출을 강화하고 싶다면 풀 크기를 더 작게 (예: 2~4) 잡거나, `CountDownLatch`를 끼워 시작 정렬을 추가하면 된다. 다만 후자의 경우 단일 Latch나 3-Latch 방식과 본질적으로 같아진다.

## 폐회로 검증 — race가 사라지는지도 같이 본다

세 방법 모두 "race를 재현했다"로 끝내면 동시성 테스트가 절반만 동작한 상태에 머문다. **fix를 적용했을 때 race가 사라지는지**까지 확인해야 테스트가 한 사이클 완성된다. 같은 3-Latch 구조에서 카운터만 `SafeCounter`로 바꾸면 다음과 같은 결과가 나온다.

```java
@Test
void safe_counter_with_three_latch_passes() throws InterruptedException {
    SafeCounter counter = new SafeCounter();   // synchronized 버전
    int threadCount = 30;
    // ... 위 3-Latch와 동일한 구조 ...

    assertThat(counter.get()).isEqualTo(threadCount);
}
```

5회 모두 `actual=30`으로 통과했다.

## 정리

| 방법 | 코드 복잡도 | 시작 정렬 정밀도 | 평균 lost (Counter 30번) | 적합한 상황 |
|---|---|---|---|---|
| ExecutorService + 단일 Latch | 낮음 | 보통 | ~12.6 | 일반적인 race 재현·회귀 테스트 |
| 3-Latch | 높음 | 가장 높음 | ~10.8 | 정밀한 정렬이 필요한 시나리오, DB 트랜잭션 검증 |
| CompletableFuture | 가장 낮음 | 없음 | 0 (시나리오에 따라 다름) | 멱등 연산의 병렬 실행, 부하 테스트 |

### 어떤 방법을 언제 쓸까

- **빠르게 race를 확인하고 싶다 → 단일 Latch.** 보일러플레이트가 적고 의도가 한눈에 들어온다. 일반적인 lost update 재현에는 이 한 가지로 충분한 경우가 많다.
- **정밀한 시작 정렬이 필요하다 → 3-Latch.** 좁은 임계영역, DB 트랜잭션, 외부 호출 등 ms 단위 동작을 검증할 때 도착 확인 단계가 가치를 발휘한다.
- **race 검출보다는 병렬 실행이 목적이다 → CompletableFuture.** 멱등 연산을 여러 개 돌리거나 가벼운 부하 테스트를 짤 때 가장 깔끔하다.
- **회귀 방어가 필요하다 → 3-Latch + `@RepeatedTest`.** 동시성 테스트는 본질적으로 비결정적이므로 한 번의 통과로 끝내지 않는다.

## 마치며

세 가지 방법은 정답이 따로 있지 않고, 검증하려는 임계영역의 성격에 맞춰 골라 쓰는 도구에 가깝다. 단일 Latch는 빠르게 race를 확인하기에 좋고, 3-Latch는 정밀한 정렬이 필요한 자리에 어울리며, CompletableFuture는 가벼운 시나리오에 잘 맞는다. 동시성 테스트를 짤 일이 생기면 이 세 가지를 일단 머릿속 서랍에 넣어두면 쓸모가 많다.
